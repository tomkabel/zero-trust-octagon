# architecture
- For axiom-to-dimension mapping: use direct mapping (each axiom explicitly declares its target dimension(s)) rather than thematic cluster mapping, to support auditability and falsifiability of the Octagon validation instrument. Confidence: 0.70
- Chapter dependencies must be strictly upstream-only — no same-layer cross-references. All cross-references must point to earlier layers, not just earlier positions within the same layer, to guarantee an acyclic dependency graph across all 23 files. Confidence: 0.70
- Archetype attack traces (Chapters 8-11) cannot introduce local dimension values — every dimension value must be defined in the morphological matrix (Chapters 4-6) first. If a trace requires a new value, extend the matrix before referencing it in the trace. The matrix is the single source of truth. Confidence: 0.70
- Use inline citation markers (e.g., `[DR-§4.2]` for domain research, `[TR-§2]` for technical research) at point-of-claim to maintain research provenance traceability, supporting independent verifiability per success metric SM-3. Confidence: 0.70
- For the Trickle-Truth Garden novelty claim: include an explicit prior art section that names and credits each prior-art component (honeypots, canary tokens, MTD, DARPA), then clearly demarcate what the Garden adds (adaptive garbage pollution rate, temporal manipulation, state-anchored deterministic synthesis, real-time TTP classification, EBK). This strengthens the novelty claim by showing exactly where the boundary is rather than making an undefended assertion. Confidence: 0.70
- Glossary consistency should be enforced through a validation gate (Appendix B checklist item): "Do all domain terms in this chapter appear in the glossary with consistent definitions?" — rather than requiring chapters to never introduce terms or forcing a central glossary as sole source of truth. The validation checklist already exists as a quality instrument; adding a glossary-specific item integrates with existing architecture. Confidence: 0.70
- Use inline markdown sections (not frontmatter) for cross-references: Prerequisites and Cross-References (Next, Builds On, Related) as standard markdown sections within chapter bodies using relative links. This matches existing chapter style, avoids frontmatter/inline duplication risk, and the acyclic dependency check can still be done by parsing heading structure. Confidence: 0.50
- For the 12-question self-assessment diagnostic routing: use truth-table validation — define a 12-row truth table for each archetype (the "pure" answer pattern for each), with tiebreakers for B/C and C/D boundaries documenting velocity and budget as decisive dimensions. This is deterministic and auditable, unlike weighted scoring functions whose weighting is less transparent. Confidence: 0.50
- Implementation pathway gates must include an escape-hatch back to Chapter 13 self-assessment on every gate failure — standard gates describe ideal progression, but every gate also documents a failure pivot with "Other Failure: return to self-assessment." This preserves routing integrity and the self-assessment's role as the architectural anchor, rather than siloing failure modes in the Aspirant's Gate chapter alone. Confidence: 0.50
- For Chapter 18's forward-looking axioms (Hendecagon/Tridecagon extensions): use a peer-review narrative rather than bare definitions or full formalization. Tell the refinement story (Octagon → Hendecagon → Tridecagon), presenting each addition with its adversarial stress-test motivation, the gap it closes, and the editorial rationale for why it wasn't included in the core eight. This matches the addendum's existing §C.1 content. Confidence: 0.70
- Every chapter must include a **Prerequisites:** section immediately after the chapter title, listing prerequisite chapters separated by commas (or "None" for Chapter 1). This is mandatory per FR-22 and ensures the chapter dependency graph is explicit and machine-verifiable at the chapter level, not just implied by cross-references. Confidence: 0.50
